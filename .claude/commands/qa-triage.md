# QA Triage — Post-Deploy Regression Detector

## Purpose

After a deploy or code change, run the full test suite and answer: **what broke, why, and what do we do about it?**

This skill doesn't write tests — it runs existing tests, diffs results against the last known state from `reports/QA_WRITE_LOG.md`, and categorizes every failure so you know which ones need attention vs which are already known.

**This is the ONLY skill that runs tests and parses results.** Other skills write tests (`/qa-write`) or analyze coverage (`/qa-crawl`), but only `/qa-triage` executes the test suite, parses output, and updates the journal with actual pass/fail data. This single-owner model prevents conflicting result interpretations.

After running tests, `/qa-triage` writes:
1. `reports/QA_TRIAGE_REPORT.md` — categorized results (regression/known/fixed/flaky)
2. Updates `reports/QA_WRITE_LOG.md` — Passing column with actual counts
3. Updates `reports/playwright-results.json` — already produced by Playwright's JSON reporter (skip on no-browser projects)
4. Updates `reports/.autopilot-state.md` — test counts (FE + BE separately)

**After deploy:** `/qa-triage` — run everything, diff against last known state.
**Specific area:** `/qa-triage /dashboard` — run tests for one page/feature only.
**Quick check:** `/qa-triage --quick` — run only L1 smoke tests for a fast pass/fail signal.

## Procedure

### Step 0: Load Last Known State

Read `reports/QA_WRITE_LOG.md`:
- **Page Status table** — which pages had tests, what was passing
- **Failing Tests table** — tests that were already failing (known bugs)
- **Flaky Tests table** — tests known to be intermittent

If `reports/QA_WRITE_LOG.md` doesn't exist: "No test history — run `/qa-write` first to establish a baseline." Still run the tests, but skip the diff (everything is "new").

### Step 1: Run Tests

Run the full test suite using JSON reporters. **NEVER parse text/list output — it truncates and causes undercounting.**

**Project-specific — skip E2E on no-browser projects.** If memory notes the project disables browser/Playwright (e.g. CDP's *No Browser Interaction*), do not run E2E. Skip it and say so in the report rather than trying to launch Playwright.

**E2E tests:**
```bash
# Always use JSON reporter — text output truncation caused 36 failures to be missed in a prior incident
npx playwright test --reporter json 2>&1 > playwright-results.json
```

Parse `playwright-results.json` to get exact pass/fail counts. The JSON has:
- `config.projects[].testDir` — which projects ran
- `suites[].suites[].specs[].tests[].results[].status` — pass/fail/skip per test
- Total counts in the summary object

**Backend tests:**
```bash
cd tests_backend && npx vitest run --reporter json 2>&1 > vitest-results.json
```

Parse `vitest-results.json` for:
- `testResults[].assertionResults[].status` — passed/failed
- `numPassedTests`, `numFailedTests` — summary counts

**If JSON reporter is unavailable:** Fall back to `--reporter list` but count BOTH pass AND fail lines explicitly. Never use `tail` on test output — failures print before passes and get truncated.

For `--quick` mode: `npx playwright test --grep "@generated by /qa-write L1" --reporter json`

### Step 2: Diff Against Last Known State

For each test result, categorize:

| Category | Definition | Action |
|----------|-----------|--------|
| **Regression** | Was passing, now failing | HIGH PRIORITY — new bug introduced |
| **Known bug** | Was already failing (in Failing Tests table) | No action — already documented |
| **Fixed** | Was failing, now passing | GOOD NEWS — remove from Failing Tests, verify intentional |
| **Bug fixed** | Was failing with `// BUG-{N}` tag, now passing | Update bugs.md — mark bug as fixed. Remove `// BUG-{N}` comment from test. |
| **Flaky** | Known flaky test that flipped again | Low priority — update flip count |
| **New failure** | Test didn't exist in last run | Investigate — could be a new test that found a real bug |
| **Stable pass** | Was passing, still passing | No action |

### Step 2.5: Re-Run Ambiguous Failures

Before categorizing failures as regressions, distinguish **null/empty results** from **wrong values**:

| Failure shape | Likely cause | Action |
|--------------|-------------|--------|
| Expected 3, got `null` / `undefined` | Async materialization not ready | Re-run 1-2x to confirm |
| Expected 3, got `0` or `5` | Compute logic bug or data mismatch | Real failure — categorize immediately |
| HTTP 500 on a previously-200 endpoint | Backend error | Real failure — categorize immediately |

Re-run null/empty failures once. Pass on re-run = **Flaky (timing)**. Fail again with same null = escalate. Fail with consistent wrong value = real regression.

### Step 2.7: Detect Bug Fixes via Passing Tests

Tests tagged with `// BUG-{N}` are expected to fail (they test correct behavior that a known bug breaks). When they start passing, the bug is fixed.

**Procedure:**
1. Grep test files for `// BUG-\d+` comments to build a bug-to-test map
2. For each tagged test, check this run's results:
   - **Still failing** → bug still exists, no action
   - **Now passing** → bug is likely fixed. Categorize as **Bug fixed**. Update `bugs.md` to mark `BUG-{N}` as fixed (add date). Remove `// BUG-{N}` comment from the test file. Report as good news.
3. Cross-check: if `bugs.md` has a bug marked fixed but the tagged test is still failing → bug is NOT actually fixed, revert the bugs.md status

**This is the mechanism.** No skips, no manual checks. Tests run every time, and the diff tells you when bugs are resolved.

### Step 3: Investigate Regressions

For each **Regression** (was passing, now failing):

1. **Check git log** — what changed since the last qa-write run?
   ```bash
   git log --oneline --since="{last run date}" -- .
   ```
2. **Identify the likely cause** — which commit touches the file/feature that the test covers?
3. **Verify it's not flaky** — if the test has flipped before (check Flaky Tests table), re-run it once to confirm
4. **Document** — add to bugs.md if it's a real regression. Include **complete reproduction curls** with exact payloads, headers, and actual responses. No prose steps. See CLAUDE.md "Bug reproduction curl requirements" for the template.

### Step 4: Write Triage Report (MUST complete before Step 7)

**CRITICAL: Write files BEFORE reporting to the user.** Steps 4 and 5 produce the persistent state that `/qa-next` reads. If you skip them and only print to the user, the triage is invisible to future conversations. The file write IS the deliverable, not the chat message.

Write `reports/QA_TRIAGE_REPORT.md` — see `references/qa-output-templates.md` § QA_TRIAGE_REPORT.md for template.

### Step 5: Update Journal

Update `reports/QA_WRITE_LOG.md` — see `references/qa-output-templates.md` § QA_WRITE_LOG.md for journal entry format. Update Failing Tests table (add regressions, remove fixed), Flaky Tests table (update flip counts), and append a Run History triage entry.

### Step 6: Visual Regression Check

Compare current `page_crawl/{page}.png` against `page_crawl/.cache/{page}_prev.png`. Record layout shifts, missing elements, broken images in the triage report. Ignore dynamic data changes. Skip if no previous screenshots exist.

### Step 7: Report to User

```
## Triage Complete

✓ {pass}/{total} tests passing
⚠ {N} regressions found — need attention
✓ {N} previously-failing tests now pass
~ {N} flaky tests flipped

Top regressions:
1. {test} — likely caused by {commit}
2. ...

Full report: reports/QA_TRIAGE_REPORT.md

### What's Next?
1. **`/qa-write --resync`** — fix broken selectors if regressions are from UI changes *(recommended if regressions are selector-related)*
2. **`/qa-write {page}`** — write new tests if regressions revealed untested paths *(recommended if regressions exposed gaps)*
3. **`/website-crawl {page}` + `/qa-crawl {page}`** — re-crawl and re-audit the page with most regressions *(if the page changed significantly)*
```

## Execution Model

The triage skill runs as a **single agent** (not per-page sub-agents) because:
- It runs the test suite once (not per-page)
- The diff logic needs the full picture across all pages
- It's fast — mostly running tests + reading files, no Playwright navigation

**Model:** Opus — regression analysis requires judgment about which commits caused which failures.

## Rules

1. **Never fix tests during triage.** Triage is read-only. Document, categorize, report. Use `/qa-write --resync` to fix broken selectors.
2. **Always diff against last known state.** Running tests without context is useless — the value is in knowing what CHANGED.
3. **Regressions are highest priority.** Report them first, investigate them first.
4. **Verify "fixed" tests.** A test that was failing and now passes might be a false positive (e.g., the test was accidentally neutered, or a retry masked the failure).
5. **Update the journal.** Every triage run must update reports/QA_WRITE_LOG.md so the next run has accurate history.
6. **Generate dashboard at the end.** After the report, run `npm run report:dashboard` (or equivalent) to produce an HTML overview. Announce the file path.
7. **JSON reporter mandatory.** See `references/qa-shared-rules.md` § JSON Reporter Rule.
8. **CI/CD integration.** If CI test results exist (`--ci-output path/to/results.json`), read those instead of re-running locally.
